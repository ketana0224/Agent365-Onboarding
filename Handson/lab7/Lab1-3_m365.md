## 0. このラボの全体像

Lab1-2 で作った 2 実体（実行体 ACA + Agent ID）に対し、**M365 のメッセージング経路**を足す。

| やること | 目的 | 本ラボの節 |
|---|---|---|
| `/api/messages`（Bot プロトコル）の実装 | Teams/M365 からのメッセージを受ける口を実行体に作る | §3 |
| メッセージング エンドポイント登録 | そのエンドポイントを Agent 365 プラットフォームに登録する | §4 |
| Developer Portal 構成 + インスタンス要求/承認 | Teams で検索・追加し、実ターンを流せる状態にする | §5 |

> **「実行確認」の2軸**: Lab1-2 §7 の **(A) Agent ID＋統制** は素の `a365 setup all` で確認できる。本章の **(B) Teams 実メッセージ往復** には、メッセージング エンドポイントの登録（`--m365` 経路）と `/api/messages` 実装が要る。`--m365` が効くのは (B) のみで、(A) には影響しない。

## 2. `--m365` と「後付け」の関係

メッセージング エンドポイントは **デプロイ後にしか URL が決まらない post-deploy 成果物**。したがって、登録のタイミングは 2 通りある。

| 方法 | コマンド | いつ |
|---|---|---|
| **A. setup 時に M365 化** | `a365 setup all --agent-name custom-maf-agent-a365 --m365 [--messaging-endpoint <url>]` | 最初から M365 扱いにする。URL 未確定なら **deferred**（後で登録）になるだけ |
| **B. 後付け登録**（本ラボ推奨） | `a365 setup blueprint --endpoint-only --agent-name custom-maf-agent-a365 --messaging-endpoint <url>` | Lab1-2 を素の `setup all` で済ませた後、デプロイ済み URL を**後から**登録 |

> `--endpoint-only` は内部で **`--m365` と同じ M365（Teams Graph）経路を自動で使う**ため、後付けでも `--m365` を別途付ける必要はない（[CLI リファレンス](https://learn.microsoft.com/microsoft-agent-365/developer/reference/cli/setup#setup-blueprint)）。
> Lab1-2 を素の `a365 setup all` で実施しているので、本ラボは **B（後付け）** で進める。

## 3. `/api/messages`（メッセージング ホスト）の実装

> **前提（既知のギャップ）**: [agent-custom-MAF-ACA-A365](agent-custom-MAF-ACA-A365/) は現状 `/chat`・`/healthz` のみを公開し、**Agent 365 のメッセージング エンドポイント `/api/messages` は未実装**。Teams 往復には、Agent 365 SDK のメッセージング ホストを組み込む必要がある。

実装の参考は、同梱の **messaging-host 参考実装** [contoso-helpdesk-a365](contoso-helpdesk-a365/) を使う（Microsoft 365 Agents SDK ホスティング + `@AGENT_APP.activity("message", auth_handlers=["AGENTIC"])` で `TurnContext` を受ける構成）。

| 観点 | 本ラボ実行体（MAF/ACA） | 参考実装（contoso-helpdesk-a365） |
|---|---|---|
| 受信口 | `/chat`（独自 JSON）のみ | `/api/messages`（Bot プロトコル） |
| 認証ハンドラ | 無し（MI で Foundry を呼ぶだけ） | `AGENTIC`（実ターンで `exchange_token`） |
| ホスティング | FastAPI + uvicorn | aiohttp（M365 Agents SDK） |

> 実装方針: 既存 MAF エージェント（`agent.py`）のロジックは残しつつ、入口を `/api/messages` の Bot プロトコル受信に差し替える。詳細手順は参考実装の [host_agent_server.py](contoso-helpdesk-a365/host_agent_server.py) を参照。
> 本ラボの主目的（**到達性の確立**）にフォーカスし、実装が完了したら §3.x として検証ログをここへ追記する。

## 4. メッセージング エンドポイントの登録（後付け）

`/api/messages` を実装してデプロイ（Lab1-2 §3.2 と同じ `deploy-aca.ps1`）した後、エンドポイントを登録する。

```powershell
# リポジトリ ルートで実行
a365 setup blueprint --endpoint-only `
  --agent-name custom-maf-agent-a365 `
  --messaging-endpoint "https://custom-maf-agent-a365.proudflower-d41f2cf1.eastus2.azurecontainerapps.io/api/messages"
```

- `Registered successfully` が出れば Blueprint `bec873dd…` にエンドポイントが紐づく。
- 設定は `a365.generated.config.json` / `.env` に同期される。
- やり直す場合は `--update-endpoint <url>`（既存を削除して再登録）。

> **ACA 構成の差異**: `a365 setup` の既定は App Service Web App（`…azurewebsites.net/api/messages`）を想定するが、本ラボは ACA で動くため、登録 URL には **ACA の FQDN** を使う。

## 5. Developer Portal 構成 → インスタンス要求・承認

> 出典: [Create an agent instance](https://learn.microsoft.com/microsoft-agent-365/developer/create-instance)

### 5.1 Developer Portal でエージェントを構成

1. [Teams Developer Portal](https://dev.teams.microsoft.com/tools/agent-blueprint/bec873dd-2082-4119-8154-e5c2edbaa48c/configuration) を開く（URL の GUID は Blueprint appId）。
2. **Agent Type = API Based** を選択。
3. **Notification URL** に §4 で登録した ACA の `/api/messages` を設定:
   `https://custom-maf-agent-a365.proudflower-d41f2cf1.eastus2.azurecontainerapps.io/api/messages`
4. **Save**（`Saved successfully`）→ **5〜10 分**反映待ち。

### 5.2 Teams で追加 → インスタンス要求

5. Teams 左レール → **Apps** で `custom-maf-agent-a365` を検索（5〜10 分で出現）。
6. ヒットしたエージェントを **Add** → **Request Instance**。
7. 管理者が [admin.cloud.microsoft/#/agents/all/requested](https://admin.cloud.microsoft/#/agents/all/requested) で **要求を承認**。
   - ボタンが効かない場合の根本原因＝**テナントで Agent 365 Frontier が未有効**。管理者に有効化を依頼。

## 6. 実ターン検証（Teams 往復）

1. 追加したエージェントとのチャットで質問を送る（例: `返品ポリシーを教えて`）。
2. 応答が返れば `/api/messages` の 1 往復が成立。
   - 失敗時の切り分け: `az containerapp logs show -n custom-maf-agent-a365 -g rg-foundryobs-eastus2 --tail 60 --type console`
3. 承認後のインスタンスは **Agent Identity（`3cdf5ac9…`）** を主体として動作し、Entra のサインイン ログにサービス プリンシパル サインインが記録される（= Lab1-2 §7 の統制対象がそのまま効く）。
